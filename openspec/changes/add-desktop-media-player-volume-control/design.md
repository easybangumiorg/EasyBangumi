## Context

当前桌面端播放器具备播放、进度与全屏等能力，但缺少稳定的音量控制闭环，用户只能依赖系统音量或外部手段调节。该变更涉及桌面端 UI 控件、ViewModel 状态管理以及 `PlayerBridge` 到 VLCJ 的音量接口联动，属于跨模块能力补齐。约束条件是：不改变 Android 行为、不引入新依赖、保持现有播放器控制结构。

## Goals / Non-Goals

**Goals:**
- 提供桌面端播放器内可视化音量调节（0-100）与静音切换。
- 让快捷交互（快捷键/滚轮）与 UI 控件走同一条状态更新链路，避免状态分叉。
- 在同一设备记住上次音量设置，重开播放器后自动恢复。
- 保证 UI 显示音量、静音状态与底层实际播放音量一致。

**Non-Goals:**
- 不实现系统级全局媒体键映射。
- 不改动 Android/iOS 端播放控制逻辑。
- 不新增均衡器、声道增强等高级音频处理能力。

## Decisions

1. 采用统一音量状态源（VM 持有 `volume` 与 `mute`），所有入口（`Seekbar`、按钮、快捷键、滚轮）只调用同一组 Action。
   - 原因：避免多入口各自写底层接口导致 UI 与实际播放状态漂移。
   - 备选方案：各入口直接调用 `PlayerBridge`；放弃原因是维护成本高且难以保证一致性。

2. 以 `PlayerBridge` 暴露标准化音量接口（读取当前音量、设置音量、设置静音），由桌面 VLCJ 实现具体映射。
   - 原因：保持 common 层与具体播放器实现解耦，后续替换播放器核心时无需改 UI 业务。
   - 备选方案：VM 直接依赖 VLCJ 细节；放弃原因是破坏模块边界。

3. 音量持久化使用现有偏好存储，保存最近一次非临时调节结果；静音状态不跨会话强制恢复（仅恢复音量值）。
   - 原因：用户更关注音量级别延续，静音跨会话恢复可能导致“无声播放”困惑。
   - 备选方案：同时持久化静音；放弃原因是启动可预期性较差。

4. 快捷交互采用固定步进（例如 5%）并限制在 [0, 100] 区间，越界时钳制。
   - 原因：保证交互反馈稳定且便于测试。
   - 备选方案：动态步进；放弃原因是学习成本高、行为不一致。

5. 桌面播放器控制栏仅常驻音量图标，鼠标悬停图标区域时在其上方展示临时音量小窗；小窗采用桌面专用竖向 `Seekbar` 样式，方向为底部 0%、顶部 100%。
   - 原因：降低控制栏常驻占位，保持交互聚焦，同时保留统一 `Seekbar` 行为链路。
   - 备选方案：常驻水平 `Slider`；放弃原因是占用控制栏宽度且与项目内统一轨道组件不一致。

6. 音量百分比文本固定展示在竖向滑轨上方，便于在悬停与拖动阶段持续感知当前值。
   - 原因：文本与交互目标同区域聚合，减少视线跳转。
   - 备选方案：文本叠加在轨道内部；放弃原因是窄轨道下可读性受限。

7. 音量轨道拖动采用“实时预览写入 + 节流持久化 + 手势结束立即提交最终值”。
   - 原因：拖动时反馈及时，同时避免高频写偏好导致的性能抖动。
   - 备选方案：仅手势结束时写入播放器与持久化；放弃原因是拖动期间反馈滞后。

## Risks / Trade-offs

- [风险] VLCJ 音量接口返回值与 UI 期望范围不完全一致 -> [缓解] 在桥接层统一做区间归一化与钳制。
- [风险] 高频滚轮事件触发过多设置调用影响性能 -> [缓解] 在输入层做节流/合并更新。
- [风险] 外部播放器或特殊流媒体场景不支持音量写入 -> [缓解] 回退为 UI 可见但禁用交互并给出状态提示。
- [取舍] 不持久化静音状态提高启动可预期性，但会牺牲少量“继续静音播放”的便利性。

## Migration Plan

1. 先扩展 `libplayer/api` 音量接口并在 desktop VLCJ 实现。
2. 在桌面播放器 VM 增加音量状态与统一 Action，接入持久化。
3. 在桌面播放器 UI 增加音量滑块、静音按钮与快捷交互绑定。
4. 通过桌面手工回归验证基础播放、切集、全屏切换下的音量一致性。
5. 若出现问题，回滚至不展示音量控件版本，同时保留接口代码以便后续修复重开。
